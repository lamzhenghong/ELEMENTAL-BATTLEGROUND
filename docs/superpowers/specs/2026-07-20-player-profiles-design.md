# Player Profiles Design

## Goal

Add a cloud-backed player identity to every Supabase account. New players choose a unique username during account creation, while the database generates an immutable player ID that can later support friend discovery without coupling identity to game-save data.

## Identity Rules

- Supabase Auth remains the source of truth for account authentication.
- One normalized email maps to one Supabase Auth user. Reusing an email must not create another account.
- Each Auth user owns exactly one `player_profiles` row through a primary-key reference to `auth.users.id` with cascade deletion.
- Usernames are required, 3-20 characters, and may contain ASCII letters, numbers, and underscores.
- Username uniqueness is case-insensitive, so `AetherHero` and `aetherhero` conflict.
- Player IDs are generated by PostgreSQL, immutable, globally unique, and displayed in the format `AETH-XXXXXXXXXXXX`.
- Existing Auth users are backfilled with a unique temporary username derived from their Auth UUID and receive a generated player ID.
- Username editing and friend features are outside this release.

## Database Design

Create `public.player_profiles` with:

- `user_id uuid primary key references auth.users(id) on delete cascade`
- `username text not null`
- `public_id text not null unique`
- `created_at timestamptz not null`
- `updated_at timestamptz not null`

Add a unique index on `lower(username)` and database checks for username format and whitespace. A signup trigger reads the validated username from `raw_user_meta_data`, generates the public player ID, and inserts the profile in the same transaction as account creation. The trigger function uses an empty search path and narrowly scoped privileges.

The public profile table is explicitly granted only the API privileges the client requires. Row Level Security allows authenticated users to read their own complete profile. A narrowly scoped username-availability RPC returns only a boolean to anonymous or authenticated callers, preventing profile enumeration while providing friendly duplicate-username feedback. Database uniqueness remains the final race-condition guard.

## Account Creation

The Create Account form adds a username field before email. Client validation runs before the request, followed by the availability RPC. The signup request sends the normalized username in Auth metadata so the database trigger can create the profile even when email confirmation means no session is returned immediately.

Duplicate usernames show `That username is already taken.` Invalid usernames receive a precise format message. Duplicate-email behavior remains controlled by Supabase Auth: no second account is created, and the UI uses a neutral confirmation message when Supabase deliberately obscures whether an email already exists.

## Client Profile Flow

After Auth resolves a session, the cloud-account hook fetches the signed-in user's profile and exposes:

- `profile.username`
- `profile.publicId`
- profile loading or error state

Signing out clears profile state. A missing profile is treated as a recoverable account error rather than silently inventing client-side identity data.

## UI

The main-menu Cloud Account card displays, with visible spacing:

1. Username
2. Email

Guest users retain the current sign-in prompt.

The signed-in Cloud Account modal displays:

- Username
- Player ID
- Email
- Cloud synchronization status and actions

The layout remains responsive, avoids horizontal overflow, and allows long email addresses to wrap or truncate safely on mobile.

## Testing And Verification

- Unit tests for username normalization and validation.
- Repository/data-source tests for profile mapping and availability checks.
- Migration contract tests for unique constraints, trigger security, explicit grants, RLS, and backfill.
- UI contract tests for username input, player ID display, username-before-email ordering, and responsive modal behavior.
- Live database tests for duplicate username rejection, case-insensitive conflicts, generated player IDs, one profile per user, duplicate-email account prevention, RLS isolation, and cleanup of temporary accounts.
- Full project tests, TypeScript check, production build, desktop/mobile browser verification, GitHub publication, and Vercel production verification.

## Security Notes

- Auth metadata is used only as signup input, never for authorization.
- Authorization uses `auth.uid()` and the profile's `user_id`.
- No service-role or secret key is exposed to the browser.
- Username availability reveals only whether a submitted exact candidate is available.
- Email uniqueness and duplicate-email obfuscation remain Supabase Auth responsibilities.
